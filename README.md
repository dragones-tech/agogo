# Agogo

Compact web server in **pure Go**: server-rendered HTML (for SEO) and a JSON
API, from a single source of truth. No frameworks or "all-in-one" libraries —
just the standard library and niche tools.

> For the **why** behind the design (principles, domain shapes, methodology),
> see [ARCHITECTURE.md](ARCHITECTURE.md).

## Philosophy

- **Only the essentials.** HTTP server, templates, and JSON come from Go's
  stdlib (`net/http`, `html/template`, `encoding/json`). Zero web framework.
- **Niche tools, not generic ones.** The data layer uses the *Repository*
  pattern: each method is exactly one query the project needs. The queries are
  generated by [sqlc](https://sqlc.dev) from real SQL — **one typed function
  per query**, no generic `.where()/.order()` and no ORM magic.
- **Minimal container.** Compiles to a static binary (`CGO_ENABLED=0`) that runs
  on a `scratch` image (~17 MB), as a non-root user, with no OS inside.

> An honest note about dependencies: the server/SEO/API is pure stdlib. The only
> component with dependencies is the `modernc.org/sqlite` driver (SQLite in pure
> Go, needed to keep the static binary and the `scratch` image); it drags in a
> few indirect dependencies. That is the minimum price of having SQLite without
> `cgo`.

## Why Agogo

- **A single binary, zero surprises.** Compiles to a static executable that runs
  on a `scratch` image (~20 MB), as a non-root user, with no OS or runtime. It
  boots in milliseconds.
- **No framework, just stdlib.** Less attack surface, no runtime "magic", easy to
  audit. The only dependency is the SQLite driver (pure Go).
- **Modular: you pay only for what you use.** You wire in `auth`, `oauth`, `logs`,
  `blog`… with one line; what you don't wire in never enters the binary. Your
  `main.go` is the list of what your app *is*.
- **SEO + API from a single source.** Server-rendered HTML (with JSON-LD and a
  sitemap) and a JSON API with OpenAPI, from the same data model.
- **Secure by default.** CSRF on forms, security headers, signed sessions (HMAC),
  parameterized SQL (sqlc), and PBKDF2 hashing — all stdlib.
- **Type-safe and yours.** No ORM, no black boxes: the code that runs is the code
  you read. Extending the server = writing a `module.go`.
- **Trivial deployment.** A binary or a minimal image; nothing to install on the
  server.

## Frontend: the same philosophy across all three layers

The frontend is **your choice** — agogo serves HTML and doesn't tie you to
anything. But out of the box it uses a sibling trio that shares exactly the same
DNA (no build, no magic, native primitives, "what you write is what runs"), one
layer for each part of the web:

| Layer | Project | What it does |
|-------|---------|--------------|
| Server (Go) | **[agogo](https://github.com/dragones-tech/agogo)** | Server-rendered HTML + JSON API from one source |
| Style (CSS) | **[scc](https://github.com/dragones-tech/scc)** | Semantic CSS: classes say *what it is*, not *how it looks* |
| UI (JS) | **[lumen](https://github.com/dragones-tech/lumen)** | Vanilla-JS UI, no build, native `<template>` |

scc and lumen are **worked on in their own repos**, not here: in agogo they are a
**vendored** copy in `internal/site/static/` (served from `'self'`, no CDN,
strict CSP — just like Swagger UI) that is **not versioned** and is re-fetched
with `./scripts/vendor-frontend.sh`. The agogo branding and the page settings
live in `static/style.css` (the vendor is not edited). See [ARCHITECTURE.md](ARCHITECTURE.md#frontend-same-philosophy-another-layer).

The examples already use them as **progressive enhancement** (the server renders
and validates the same without JS):

- **scc** styles the semantic HTML of every page (`class="card"`,
  `class="field"`, `class="alert"`, `<button data-variant="primary">`…).
- **lumen** adds client-side convenience: instant catalog filtering
  (`static/js/catalog/`) and contact-form validation mirroring the server's
  rules (`static/js/contact/`).

Swap them for whatever you want: the philosophy doesn't force you to use them,
only that, if you do, the three fit together without friction.

## Structure (package-by-feature)

```
.
├── main.go                      # the "Gemfile": creates the App and wires modules with app.Use(...)
├── sqlc.yaml                    # sqlc configuration
├── Dockerfile                   # multi-stage → scratch
├── cmd/
│   ├── migrate/                 # command: applies the schema (bootstrap)
│   └── seed/                    # command: sample data (development only)
└── internal/
    │  ── CORE + utilities (base, not modules) ──
    ├── app/                      # HOST: App, Module interface, hook points
    ├── router/                   # Express-style Router (r.Get/r.Post + middleware)
    ├── config/                   # typed and validated environment loading
    ├── session/                  # signed-cookie sessions (HMAC)
    ├── identity/                    # identity service over session + Require
    ├── password/                 # PBKDF2 hashing (stdlib)
    ├── web/                      # Resource[T]: generic DB resource
    ├── view/                     # shared HTML layout + Render
    ├── respond/                  # respond.JSON / respond.Error / respond.ServerError
    ├── validate/                 # minimal, explicit input validator (no reflection)
    ├── jsonld/                   # schema.org builders
    ├── csrf/                     # CSRF protection (double cookie)
    ├── sitemap/                  # sitemap contract (URL, Source, Entries)
    ├── middleware/               # security baseline: Recover, SecurityHeaders
    │  ── MODULES (opt-in; each with module.go → Module()) ──
    ├── logs/                     # global logging middleware
    ├── productos/  blog/         # DB resources (handler.go, module.go, sql/, db/, templates/)
    ├── paginas/                  # static pages (no DB)
    ├── contacto/                 # form (GET/POST + CSRF + flash)
    ├── auth/                    # username/password authentication (login/account, reuses identity)
    ├── oauth/                    # OAuth 2.0 authentication (reuses identity), stdlib
    ├── openapi/                  # openapi.json + vendored Swagger UI
    └── site/                     # robots.txt, sitemap.xml, /static
        └── static/
            ├── scc/              # vendor (gitignored): semantic CSS — re-fetch with scripts/vendor-frontend.sh
            ├── lumen/            # vendor (gitignored): vanilla-JS UI — same
            ├── js/               # our JS, package-by-feature (like the Go)
            │   ├── nav.js        #   partial site navigation (only swaps <main>)
            │   ├── catalog/      #   product.js (Model) · product-item.js, catalog-view.js (View) · index.js (entry)
            │   └── contact/      #   index.js (form validation)
            └── style.css         # page frame (layout) + agogo branding; the design comes from scc
```

The server is a **core (`app`) + pluggable modules**. Each module has a
`module.go` with `Module()` and registers its routes, migrations, and sitemap
sources in its `Register(*App)`. `main.go` wires them with `app.Use(...)` — that
list is the overview of what the server does. See [ARCHITECTURE.md](ARCHITECTURE.md)
for the details of the model.

### Generic resource (`web.Resource[T]`)

The 4 handlers (list/detail × HTML/JSON), the route registration, and the
sitemap contribution are written **only once** in `internal/web/resource.go`, as
a generic `Resource[T]`. A domain reimplements none of that: it only
**configures** a `Resource` with its types, queries (sqlc), templates, and SEO
metadata. Adding a common route/behavior = editing `resource.go` once and the N
domains inherit it. It's our code and type-safe (compile-time generics, no
reflection).

## Installation

Requirements: **Go 1.26+**. (Docker is optional, for the container image;
`sqlc` is optional, **only** to regenerate code after touching the SQL.)

Agogo is not a library you install with `go get`: it's a **base you clone and
make your own** (the framework packages live in `internal/`, on purpose — the
code that runs is the code you read and edit).

```bash
git clone https://github.com/dragones-tech/agogo
cd agogo
./scripts/vendor-frontend.sh   # fetches scc + lumen (not versioned here; they live in their own repos)
```

And you run it as explained below in **How to run it**.

## How to run it

### Local

The server **only serves**: it doesn't touch the schema or seed data (that lives
outside the site, as development commands). On a fresh DB:

```bash
go run ./cmd/migrate   # applies the schema (bootstrap; also used on deploy)
go run ./cmd/seed      # sample data (DEVELOPMENT ONLY; idempotent)
go run .               # starts the server → http://localhost:8888
```

In development, `go run ./cmd/seed` is enough (it ensures the schema and then
seeds), followed by `go run .`.

### In a container

```bash
docker build -t agogo .
docker run -p 8888:8888 agogo
# → http://localhost:8888
```

The DB lives in `/data` inside the container. To persist it, mount a volume:

```bash
docker run -p 8888:8888 -v agogo-data:/data agogo
```

## HTTP protocols (HTTP/1.1, HTTP/2, HTTP/3)

> Status: in testing. The binary boots today in plaintext
> (`ListenAndServe`), so it speaks **HTTP/1.1**.

| Protocol | Status | How you get it |
|----------|--------|----------------|
| **HTTP/1.1** | ✅ Active | Default in `net/http` |
| **HTTP/2** | ⚙️ One step away | Serve over TLS → `net/http` negotiates h2 via ALPN, **no dependencies** |
| **HTTP/3** | 🚫 Not in the binary | QUIC is not in Go's stdlib; it's terminated at the edge (proxy/CDN) |

### HTTP/2 inside the binary (pure stdlib)

`net/http` enables HTTP/2 **automatically over TLS** (built in since Go 1.6, zero
dependencies). It's enough to serve with certificates:

```go
// h2 is negotiated automatically via ALPN when the client supports it
srv.ListenAndServeTLS(certFile, keyFile)
```

HTTP/2 in plaintext (h2c) is **not** active by default and would require
`golang.org/x/net/http2/h2c` (a dependency), so we don't use it: HTTP/2 = TLS.

### Recommended deployment: terminate h2/h3 at the edge

The standard practice is to **not** put QUIC or TLS in the binary, but to
terminate the modern protocols at a reverse proxy / CDN and keep the app compact:

```
Client ──h3/h2/TLS──► Caddy / nginx / CDN ──h1.1 or h2──► Go binary
       (UDP + TCP)        (the edge)        (private network / socket)
```

The edge speaks **HTTP/3 + HTTP/2 + TLS** outward; the binary stays simple, with
no dependencies and without managing certificates. (The binary already
**gzip-compresses** text responses on its own —`compress/gzip`, stdlib—, so it
works fine even without a proxy; the edge adds zstd/brotli if you want.) Minimal
example with Caddy (HTTP/2 and HTTP/3 + automatic TLS, ready out of the box):

```caddyfile
# Caddyfile (deployment example, not included in the image)
agogo.com {
	encode gzip zstd
	reverse_proxy localhost:8888   # or unix//run/agogo.sock
}
```

Caddy serves HTTP/3 by default; there's no need to touch the Go code.

## Web standards

"Web standards" has two meanings and it's worth not confusing them:

- **The web's protocols and formats (HTTP, HTML5, JSON, TLS, sitemap, JSON-LD):**
  yes, fully. `net/http` implements the IETF RFCs directly, and the
  HTML/JSON/XML we emit follows the corresponding standards.
- **The "Web Platform API" (Fetch `Request`/`Response` objects), which is what JS
  frameworks like Hono call "web standards":** doesn't apply to Go. That
  abstraction provides portability across JavaScript runtimes (Workers, Deno,
  Bun). Here portability is achieved another way: **a static binary that runs on
  any Linux without a runtime**. Same goal (not tying yourself to a proprietary
  API), different path.

## Adding a query (sqlc flow)

1. Write the query in `internal/productos/sql/queries.sql`:

   ```sql
   -- name: ListDestacados :many
   SELECT slug, titulo, descripcion, precio
   FROM productos WHERE precio >= ? ORDER BY precio DESC;
   ```

2. Regenerate the typed code:

   ```bash
   # sqlc is installed with: go install github.com/sqlc-dev/sqlc/cmd/sqlc@latest
   # and it lands in Go's GOBIN (which may not be on the PATH by default):
   export PATH="$PATH:$(go env GOBIN)"
   sqlc generate
   ```

3. Use the generated function (`q.ListDestacados(ctx, 100)`) in your handler.

What you don't write doesn't exist: there is no universal query builder.

## Adding a new domain

1. Create `internal/<domain>/` with its `handler.go`, `sql/`, `templates/`.
2. Expose a `NewHandler(...)` and a `Routes(mux)`.
3. If it has public URLs, implement `site.SitemapSource` and pass it to
   `site.NewHandler(baseURL, productosH, <domainH>)` in `main.go`.
